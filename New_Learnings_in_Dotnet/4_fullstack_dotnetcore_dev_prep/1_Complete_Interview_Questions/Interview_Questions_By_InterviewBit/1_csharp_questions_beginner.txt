Q2. What is the difference between value types and reference types?

Answer: 

1). Value Types --> 
    * Value types are data types that directly store data in a fixed amount of memory.
    * They are stored on the stack, which is a region of memory that is allocated and deallocated automatically by the program.
    * Value types are passed by value, which means that when they are assigned to a variable or passed to a method, a copy of the data is created and used.

Example :- int,float,double,bool,structs,enum,char etc.

The pros and cons of value types.
    * Pros
        * They are efficient, as they use less memory and have faster access time.
        * They have predictable memory allocation, as they are automatically created and destroyed by the program.
    * Cons
        * They are limited in size, as they cannot exceed the size of the stack.
        * They can be expensive to copy, as they create a new instance of the data every time they are assigned or passed.

2). Reference Types --> 
    * Reference types are data types that store references to data in a variable amount of memory.
    * They are stored on the heap, which is a region of memory that is allocated and deallocated manually by the program (GC).
    * Reference types are passed by reference, which means that when they are assigned to a variable or passed to a method, the reference to the data is used, not the data itself.

Example :- object (stores any kind of data), arrays, strings(but immutable), classes etc.

The pros and cons of reference types.
    * Pros
        * They have dynamic memory allocation, as they can grow or shrink as needed.
        * They have flexibility, as they can store any kind of data and can be modified at runtime.
    * Cons
        * They have indirect access, as they require an extra step to access the data through the reference.
        * They require memory management, as they need to be manually created and destroyed by the program.

Key Differences between Value and Reference Types
    * Memory storage location: Value types are stored on the stack, while reference types are stored on the heap.
    * Copying behavior: Value types create a copy of the data when they are assigned or passed, while reference types use the same instance of the data when they are assigned or passed.
    * Equality comparison: Value types compare the data by value, while reference types compare the data by reference.
    * Immutability vs. mutability: Value types are immutable, which means that they cannot be changed after they are created, while reference types are mutable, which means that they can be changed after they are created.
    * Common use cases: Value types are used for simple and primitive data, such as numbers, booleans, and structs, while reference types are used for complex and dynamic data, such as objects, arrays, and classes.

Best Practices
    * Use value types for simple and primitive data that do not need to be modified or shared, and use reference types for complex and dynamic data that need to be modified or shared.
    * Use value types when the data is small and can fit on the stack, and use reference types when the data is large and needs to be allocated on the heap.
    * Use value types when the data needs to be copied and isolated, and use reference types when the data needs to be referenced and shared.
    * Use value types when the data needs to be compared by value, and use reference types when the data needs to be compared by reference.
    * Use value types when the data needs to be immutable, and use reference types when the data needs to be mutable.

Q3. How garbage collector (GC) works in background in case of value types and reference types in C# ?

Answer: 

1). Value types live on the stack, reference types live on the heap. The GC only manages the heap — it never touches the stack.
2). Stack memory for value types is managed automatically by the CPU via push/pop — zero GC involvement. Reference types on the heap are managed by  the GC using a generational mark-sweep-compact algorithm, where Gen 0 is collected most aggressively since most objects are short-lived.

Bonus Points
1). The GC handles memory, but not unmanaged resources like file handles or DB connections. For those, you implement IDisposable and use using blocks so cleanup is deterministic, not GC-dependent.
2). LOH (Large Object Heap) — Objects larger than 85KB go to a special heap that is never compacted by default (too expensive to move large chunks). This can cause memory fragmentation over time in long-running apps.
3). The LOH holds objects ≥ 85KB and is collected like the SOH (Small Object Heap) but never compacted by default — dead objects leave holes that accumulate into fragmentation. The fix is ArrayPool<T> to reuse large buffers instead of allocating new ones.

Q5. What are the types of classes in C#?

Answer: Class is an entity that encapsulates all the properties of its objects and instances as a single unit. 

C# has four types of such classes:
1). Concrete Class: The everyday class. Can be instantiated, inherited, and contains full implementations.

2). Abstract class: Abstract classes are classes that cannot be instantiated where you cannot create objects. Abstract classes work on the OOPS concept of abstraction. Abstraction helps to extract essential details and hide the unessential ones.

3). Sealed class: Sealed classes are classes that cannot be inherited. Use the keyword sealed to restrict access to users to inherit that class. 

4). Static class: Static class, defined by the keyword ‘static’ does not allow inheritance. Therefore, you cannot create an object for a static class.

5). Partial class: Partial class, defined by the keyword ‘partial’ allows you to split the definition of a single class across multiple source files. When you compile your application, the C# compiler combines all these parts into one single class

Q6. What is the difference between an abstract class and an interface?

Answer: 

1). Abstract classes are classes that cannot be instantiated.
    The interface is like an abstract class because all the methods inside the interface are abstract methods.

2). Surprisingly, abstract classes can have both abstract and non-abstract methods but all the methods of an interface are abstract methods.

3). Since abstract classes can have both abstract and non-abstract methods, we need to use the Abstract keyword to declare abstract methods. But in the interface, there is no such need.

4). An abstract class has constructors while an interface encompasses none.

Q7. What are extension methods in C#?

Answer:

It allow you to "add" new methods to existing types (like string, int, or even your own classes) without modifying their original source code. They act like "syntactic sugar," letting you call a static method as if it were a regular instance method on an object. 

Key Benefits
-------------
1). Extend Closed Types: You can add functionality to "sealed" classes (like string) that you cannot inherit from.
2). Cleaner Code: It is more readable to write myString.DoSomething() than MyHelper.DoSomething(myString).
3). LINQ Integration: Most LINQ features (like .Where() or .Select()) are actually extension methods.


Example:

public static class MyExtensions
{
    // Adding a method to the 'string' type
    public static int WordCount(this string str)
    {
        return str.Split(' ').Length;
    }
}
// Usage:
string sentence = "Hello world from C#";
int count = sentence.WordCount(); // Called like a built-in method

8. What are Generics in C#?

Answer:

A way to write code (classes, methods, or interfaces) that can work with any data type without being locked into one specifically.
Think of them as a template or a "label" that says, "I will handle a type later; for now, just call it T".

Key Benefits
-------------
1). Code Reusability: You write a single piece of code once, and it can be used with int, string, or custom objects.
2). Type Safety: The compiler catches errors if you try to put a string into something you defined as a "Generic List of Integers".
3). Performance: They avoid "boxing" and "unboxing" (converting value types to objects), making them faster than non-generic collections.

Example: Imagine a physical box. Without generics, you’d need an "AppleBox" for apples and a "ToyBox" for toys. With generics, you just have a Box<T>.

// Definition: T is a placeholder for the type
public class Box<T>
{
    public T Content { get; set; }
}

// Usage: You decide the type when you create the object
Box<int> numberBox = new Box<int>();
numberBox.Content = 10;

Box<string> textBox = new Box<string>();
textBox.Content = "Hello";

Box<List<string>> textBox = new Box<List<string>>();
textBox.Content = new List<string> { "Hello", "World", "!" };
